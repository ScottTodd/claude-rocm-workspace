* **Author:** Laura Promberger (@HereThereBeDragons)
* **Base:** `main`
* **Reviewed:** 2026-03-23 (updated; first review 2026-03-02)
* **Net changes:** +1393 lines, -508 lines across 4 files

---

## Summary

Replaces the nested dictionary format in `new_amdgpu_family_matrix.py` with two
new files using dataclasses:

- `new_amdgpu_family_matrix_types.py` — schema definitions (dataclasses, lookup
  logic, serialization)
- `new_amdgpu_family_matrix_data.py` — matrix data (arch entries, build
  variants, predefined groups, cmake family parsing)

Each GPU is now an individual `MatrixEntry` with its own target name (e.g.
`gfx942`, `gfx1030`), and family membership is auto-populated from
`cmake/therock_amdgpu_targets.cmake`. This replaces the old family→scope→platform
nesting.

Also bundles several data changes: gfx90X split into gfx906/908/90a, tsan build
variant added, `push_on_success` removed, new runner labels, cumulative group
tiers, and individual target expansion for gfx101X/103X/110X families.

---

## Overall Assessment

**⚠️ CHANGES REQUESTED** — The dataclass architecture is a clear improvement
and the test suite is thorough. However, there are data fidelity issues that
would silently change CI behavior, a bug in the cmake parser, and style guide
violations.

**Strengths:**
- Clean dataclass hierarchy with sensible defaults and good docstrings
- `TestConfig.__post_init__` for `run_tests` inference from runner presence
- `GpuRunners.extra` dict for non-standard runners without schema changes
- `_MatrixEntries` class with `isinstance` filtering — much better than the
  earlier `globals()` approach
- Auto-population of family lists from cmake avoids manual synchronization
- `lookup()` with `EntryLookupResult` gives callers resolution metadata
- Comprehensive test suite covering lookup, inference, serialization, defaults,
  deduplication, and validation

**Blocking/Important Issues:**
- Family resolution layer adds complexity without clear benefit (see architectural section)
- Bare `raise` in `_parse_amdgpu_targets()` — crashes with no useful message
- gfx1030 linux: `run_tests` silently changed from `False` to `True`
- gfx1030 windows: `expect_pytorch_failure` dropped
- gfx1152/gfx1153: `expect_failure` dropped from builds
- asan build variant: `expect_failure` dropped
- `from __future__ import annotations` violates style guide
- `Optional[T]` used instead of `T | None`

## Detailed Review

### 1. new_amdgpu_family_matrix_data.py — `_parse_amdgpu_targets()`

#### ❌ BLOCKING: Bare `raise` will crash with confusing error

`_parse_amdgpu_targets()` line ~39 (data file line 600):

```python
if "FAMILY" not in block:
    raise
```

A bare `raise` outside an `except` block raises `RuntimeError: No active
exception to reraise` — a confusing message that gives no clue what went wrong.
Per the [Python style guide's fail-fast
section](https://github.com/ROCm/TheRock/blob/main/docs/development/style_guides/python_style_guide.md#fail-fast-behavior),
error messages should explain what happened and what to do about it.

- **Required action:** Replace with a descriptive exception:
  ```python
  if "FAMILY" not in block:
      target = block.split()[0] if block.split() else "<unknown>"
      raise ValueError(
          f"therock_add_amdgpu_target({target}) missing FAMILY argument "
          f"in cmake/therock_amdgpu_targets.cmake"
      )
  ```

#### ⚠️ IMPORTANT: `_parse_amdgpu_targets()` parsing is fragile

The parser splits on `"therock_add_amdgpu_target("` and then on `"FAMILY"`, with
`split("\n")[0]` to grab the family line. This assumes:
- FAMILY args are all on one line
- No commented-out `therock_add_amdgpu_target()` calls
- No string literals containing these tokens

This works today, but cmake file changes could break it silently.

- **Recommendation:** Add a test that validates the parsed output against known
  entries (e.g., assert gfx942 has `gfx94X-dcgpu` in its family list). The
  existing `TestFamilyPopulation` tests do this — good.

### 2. new_amdgpu_family_matrix_data.py — Data Fidelity

#### ❌ BLOCKING: gfx1030 linux `run_tests` changed from False to True

Old file had explicit `run_tests: False` with TODO `#2740` (amdsmi test broken):

```python
# Old
"gfx103X": {"dgpu": {"linux": {"test": {"run_tests": False, ...}}}}
```

New `GFX1030` linux:

```python
GFX1030 = MatrixEntry(
    target="gfx1030",
    linux=PlatformConfig(
        test=TestConfig(
            runs_on=GpuRunners(test="linux-gfx1030-gpu-rocm"),
            sanity_check_only_for_family=True,
            fetch_gfx_targets=["gfx1030"],
        ),
    ),
    ...
)
```

Because a runner is set and `run_tests` is not explicitly `False`,
`TestConfig.__post_init__` infers `run_tests=True`. This is a subtle side effect
of the auto-inference that the PR calls a feature. The TODO comment about the
amdsmi issue is also lost.

- **Impact:** CI would attempt to run tests on hardware that may not be stable
- **Required action:** Add `run_tests=False` and the TODO comment. Also applies
  to `fetch_gfx_targets` which old had as `[]` (disabled) but new has as
  `["gfx1030"]`.

#### ❌ BLOCKING: gfx1030 windows `expect_pytorch_failure` dropped

Old gfx103X-dgpu windows had `expect_pytorch_failure: True` (with TODO `#3200`).
New `GFX1030` windows `TestConfig` doesn't include it:

```python
windows=PlatformConfig(
    test=TestConfig(
        run_tests=False,
        runs_on=GpuRunners(test="windows-gfx1030-gpu-rocm"),
        sanity_check_only_for_family=True,
    ),
),
```

- **Required action:** Add `expect_pytorch_failure=True` and the TODO comment.

#### ❌ BLOCKING: gfx1152 `expect_failure` silently dropped

Old file had `expect_failure: True` for both linux and windows builds. New entry
uses bare `PlatformConfig()` on both, which defaults `expect_failure` to `False`.

- **Impact:** If builds fail (they were expected to), CI treats them as unexpected
  regressions instead of expected failures
- **Required action:** Either add `build=BuildConfig(expect_failure=True)` on
  both platforms, or confirm this is intentional and document it.

#### ❌ BLOCKING: gfx1153 `expect_failure` silently dropped

Same issue as gfx1152. Old file had `expect_failure: True` on both linux and
windows builds.

- **Required action:** Same as gfx1152.

#### ❌ BLOCKING: asan build variant `expect_failure` dropped

Old file:

```python
"asan": {
    "build_variant_label": "asan",
    ...
    "expect_failure": True,
}
```

New file:

```python
"asan": BuildVariantInfo(
    label="asan",
    suffix="asan",
    cmake_preset="linux-release-asan",
),
```

No `expect_failure=True`. Note that the new `tsan` variant does have
`expect_failure=True`. If asan is now expected to pass, this should be called out
in the PR description as an intentional change.

- **Required action:** Restore `expect_failure=True` or document the intentional
  change.

#### ⚠️ IMPORTANT: Predefined groups changed from disjoint to cumulative

Old groups were non-overlapping:

```python
"amdgpu_presubmit":  ["gfx94X-dcgpu", "gfx110X-all", "gfx1151", "gfx120X-all"]
"amdgpu_postsubmit": ["gfx950-dcgpu"]
"amdgpu_nightly":    ["gfx90X-dcgpu", "gfx101X-dgpu", ...]
```

New groups are cumulative (each tier includes all previous):

```python
presubmit  = sorted(["gfx94X-dcgpu", "gfx110X-all", "gfx1151", "gfx120X-all"])
postsubmit = sorted(set(presubmit) | {"gfx950-dcgpu"})
nightly    = sorted(set(postsubmit) | {"gfx906", "gfx908", ...})
```

This is a semantic change — `amdgpu_postsubmit` now contains 5 entries instead
of 1. The code comment says "Each tier is built by unioning the previous tier"
which documents the intent, but the PR description doesn't mention this change.

Since these groups aren't consumed by any workflow yet, this isn't blocking, but
it should be documented in the PR description.

- **Recommendation:** Add a note to the PR description explaining the cumulative
  tier change.

#### ⚠️ IMPORTANT: gfx94X and gfx950 dropped Windows config

Old file had Windows config for both (release build, no tests, no release push).
New `GFX942` and `GFX950` have `windows=None` (Linux-only).

This changes the `to_dict()` output — `windows` key won't appear at all. If any
future consumer iterates platforms and expects both to be present, this would
cause a `KeyError` or `AttributeError`.

- **Recommendation:** Confirm this is intentional. If Windows builds are still
  desired (even minimal), keep `windows=PlatformConfig()`.

### 3. new_amdgpu_family_matrix_types.py — Style

#### ❌ BLOCKING: `from __future__ import annotations` violates style guide

Line 1 of the types file:

```python

```

The [Python style
guide](https://github.com/ROCm/TheRock/blob/main/docs/development/style_guides/python_style_guide.md)

  references (if any) to use string annotations or reorder definitions.

The style guide requires `T | None` syntax. Used in:
- `AllBuildVariants.get()` return type
- `TestConfig.run_tests` field type
- `MatrixEntry.linux` and `MatrixEntry.windows` field types
- `MatrixEntry.platform_config()` return type
- `MatrixEntry.to_dict()` platform parameter
- `AmdGpuFamilyMatrix.get_entry()` return type
- `AmdGpuFamilyMatrix.get_default_for_family()` return type
- `AmdGpuFamilyMatrix.lookup()` return type

- **Required action:** Replace all `Optional[X]` with `X | None`.

### 4. new_amdgpu_family_matrix_types.py — Design

#### 💡 SUGGESTION: `to_nested_dict()` name is misleading

The method produces a flat dict keyed by target name (not nested by
family/scope). `to_dict()` or `to_target_dict()` would be more accurate.

#### 💡 SUGGESTION: Consider `__repr__` for `MatrixEntry`

### 5. Tests — new_amdgpu_family_matrix_test.py

#### ⚠️ IMPORTANT: Roughly half the tests are change detectors tied to today's data

The test suite has a good set of behavioral/mechanism tests (~15), but roughly
half (~20) are change detector tests that assert specific configuration values
for specific GPU entries. These break whenever data evolves — adding a chip,
changing a runner label, flipping `expect_failure` — but the failures provide no
bug signal, just noise from data evolution.

This is particularly ironic since the PR's main selling point is "easier
maintenance when adding/modifying entries." The test suite creates friction for
exactly those operations.

**Change detector examples (not exhaustive):**

| Test | What it snapshots | Breaks when... |
|------|-------------------|----------------|
| `test_all_known_archs_present` | Hardcoded list of 22 targets | Any chip added or removed |
| `test_gfx94x_has_asan_build_variant` | gfx942's variant list | Variant added/removed for gfx942 |
| `test_gfx1151_windows_test_scope` | Value `"full"` | Scope changes |
| `test_gfx94x_sandbox_runner_in_to_dict` | Exact runner string | Runner renamed |
| `test_get_targets_for_family_gfx120X_all` | Exactly `["gfx1200", "gfx1201"]` | New gfx120X chip |
| `test_to_nested_dict_gfx1101_match_layout` | **30-line dict literal** | ANY schema change, field addition, or config tweak to gfx1101 |

**Good behavioral tests (keep these as-is):**
- `TestDefaultConfig` — tests dataclass default values
- `TestRunTestsInference` — tests `__post_init__` logic with constructed instances
- `TestToDict` field-presence checks (keys exist, not specific values)
- `test_is_family_default_validation_prevents_duplicates` (synthetic matrix)
- `test_all_keys_are_unique`, `test_predefined_groups_reference_valid_keys` (invariants)
- `test_unknown_platform_raises`, `test_unknown_variant_returns_none` (error paths)
- `test_deduplicate_*`, `test_empty_input_returns_empty_result` (mechanism)

**Recommendation:** Restructure the tests into two categories:

1. **Mechanism tests** (in this file): Test lookup, inference, serialization,
   validation using **synthetic `AmdGpuFamilyMatrix` instances** with minimal
   fake data. These never break when real data changes.

2. **Data consistency tests** (separate file or class): A small number of
   structural invariants about the real matrix — e.g., "all group keys resolve,"
   "no duplicate defaults per family," "every entry has at least one platform."
   These are cheap to maintain because they test properties, not specific values.

Delete the snapshot tests entirely. They add maintenance cost without catching
real bugs. If the concern is "did we port the data correctly," that's a one-time
migration check, not a permanent test.

#### 💡 SUGGESTION: Missing negative test for `_parse_amdgpu_targets()`

No test covers the case where `_parse_amdgpu_targets()` encounters a target
without a FAMILY arg. Given the bare `raise` bug, this should be tested once the
error handling is fixed.

## ⚠️ IMPORTANT: Family Resolution Layer Should Be Removed

The PR adds a family resolution layer that the old code didn't have:

- `_parse_amdgpu_targets()` — parses cmake to build target→family mappings
- `MatrixEntry.family` field, `is_family_default` flag, validation
- `get_default_for_family()`, `get_targets_for_family()`
- Family/prefix resolution in `get_entry()` and `lookup()`
- `EntryLookupResult` with `resolved_via` metadata (target/family/family_prefix)

This is roughly **40% of the types file** and the entire cmake parser.

**The problem isn't just that this code may be short-lived — it's that it
actively pushes the data model in the wrong direction.**

The `is_family_default` pattern encourages "one representative chip per family"
data layout. Look at the gfx110X entries: gfx1101 gets the full config (runners,
test targets, release config), while gfx1100/1102/1103 are empty shells with
bare `PlatformConfig()`. The data structure assumes that when you ask about
gfx110X, you only care about one chip's config.

This works against per-target CI, where each chip should carry its own config
independently. And the resolution layer doesn't serve the actual consumers —
`configure_ci.py` and workflow scripts are code we control. We can just use
target names directly. There's no external user calling `get_entry("gfx110X-all")`
who can't be updated to call `get_entry("gfx1101")`.

The predefined groups demonstrate this: they currently use family keys like
`"gfx94X-dcgpu"` and `"gfx110X-all"` that require the resolution layer. If they
just listed `"gfx942"` and `"gfx1101"` directly, the entire family resolution
layer becomes dead code.

**Recommendation:** Remove the family resolution layer from this PR:

| Remove | Keep |
|--------|------|
| `_parse_amdgpu_targets()` | `MatrixEntry` per-target dataclass hierarchy |
| `MatrixEntry.family`, `is_family_default` | `PlatformConfig`, `BuildConfig`, `TestConfig`, etc. |
| `_validate_is_family_default()` | `all_build_variants` |
| `get_default_for_family()`, `get_targets_for_family()` | `get_entry()` by target name only |
| Family/prefix resolution in `get_entry()` and `lookup()` | Predefined groups (re-keyed to target names) |
| `EntryLookupResult` with resolution metadata | `to_dict()` serialization |
| `cmake_families` parameter | `_MatrixEntries` auto-discovery |
| ~8 family-related tests | Behavioral/mechanism tests |

This cuts ~200 lines of types code, the entire cmake parser, and the associated
tests. It also removes the data pressure toward "one config per family" and
makes each target a first-class citizen — which is where multi-arch CI is headed.

## Conclusion

**Approval Status: ⚠️ CHANGES REQUESTED**

The dataclass architecture is a clear improvement over the nested dict — type
safety, self-documenting fields, and a clean per-target data model. The
`_MatrixEntries` class approach for auto-discovery is much better than the
earlier `globals()` iteration.

The main concerns are:

1. **Family resolution layer** — ~40% of the types code is a family indirection
   layer that the old code didn't have, that actively encourages "one chip per
   family" data layout, and that serves no purpose for internal CI consumers who
   can just use target names directly. Should be removed.
2. **Data fidelity** — several entries silently change behavior (`run_tests`,
   `expect_failure`, `expect_pytorch_failure`) due to the inference feature
   interacting with incomplete porting
3. **Bug** — bare `raise` in `_parse_amdgpu_targets()` (goes away if cmake
   parser is removed per point 1)
4. **Style** — `from __future__ import annotations` and `Optional[T]`
5. **Test quality** — roughly half the tests are change detectors that will
   create maintenance friction for the exact kind of changes the refactor is
   supposed to make easier
   builds
